# commit-push

Commit and push the current working tree to the current branch using a conventional-style commit title.

## Title format (REQUIRED)

```
<type>: <short title>
<type>: <short title> (#<issue-number>)
```

- `<type>` is one of: `wip`, `feat`, `fix`, `chore`, `refactor`, `docs`, `test`, `perf`, `build`, `ci`, `style`. Pick the one that best matches the dominant change.
- `<short title>` is imperative, lowercase, no trailing period, ≤ 60 chars.
- Append `(#<issue>)` only when an issue number is explicitly known — from the user's message, a branch name like `123-fix-foo` / `issue/123-...` / `fix/123-...`, or a referenced GitHub URL. Never fabricate one.
- Body is optional — add 1–3 bullet points only if the change is not self-evident from the diff. No "Generated by" footer, no emojis.

Examples:

- `fix: handle missing institute field in dawum payload (#1)`
- `feat: add sonntagsfrage trend chart`
- `wip: refactor api client`

## Procedure

Run these in parallel first to understand the state of the repo:

```bash
git status
git diff
git log -n 10 --oneline
git rev-parse --abbrev-ref HEAD
```

Then:

1. **Pick the type.** Look at the dominant change in the diff:
    - New capability / new file with new behaviour → `feat`
    - Bug fix → `fix`
    - Pure refactor (no behaviour change) → `refactor`
    - Docs / CLAUDE.md / README → `docs`
    - Tests only → `test`
    - Tooling, deps, CI, build config → `chore` / `build` / `ci`
    - Mixed and incomplete → `wip`

2. **Determine the issue number.** Check, in order:
    - Explicit `#N` in the user's most recent message.
    - Current branch name (`123-...`, `issue/123-...`, `fix/123-...`, `feat/123-...`).
    - URL of the form `…/issues/N` mentioned in the conversation.

    If none found, omit `(#N)` — do not invent one.

3. **Stage explicitly.** Use `git add <files>` for each changed/new file from `git status`. Do **not** use `git add -A` or `git add .` (avoids accidentally committing `.env`, secrets, build artefacts under `.next/`/`dist/`/`build/`). Skip anything under `node_modules/`, `.next/`, `dist/`, `build/`, `.env*`, `data/`, `logs/`.

4. **Commit.** Use a HEREDOC so the message formats correctly:

    ```bash
    git commit -m "$(cat <<'EOF'
    <type>: <short title> (#<issue>)
    EOF
    )"
    ```

    With body (only when useful):

    ```bash
    git commit -m "$(cat <<'EOF'
    <type>: <short title> (#<issue>)

    - point 1
    - point 2
    EOF
    )"
    ```

    Never use `--amend`, `--no-verify`, or `-c commit.gpgsign=false`.

5. **Push.** Push to the current branch's upstream:

    ```bash
    git push
    ```

    If `git push` reports "no upstream branch", set it for the current branch:

    ```bash
    git push -u origin "$(git rev-parse --abbrev-ref HEAD)"
    ```

    Never force-push. Never push to `main` if the current branch is something else. If the current branch IS `main`, that is fine for solo/personal repos; for shared repos confirm with the user.

6. **Verify.** Run `git status` and a final `git log -n 1 --oneline` and report the new commit hash + subject + remote ref to the user in one line.

## Refuse / pause if

- The diff contains an `.env`, credentials, API keys, or anything under `data/`.
- The diff spans changes that look unrelated to a single intent — split into multiple commits (each its own conventional type). Only ask the user when the split is ambiguous.
- Branch is a shared collaboration branch you do not own — confirm with the user before committing.
- Pre-commit / pre-push hooks fail — fix the underlying problem, re-stage, and create a **new** commit. Never bypass with `--no-verify`.

## Autonomous invocation

In addition to explicit user requests (`/commit-push`, "commit", "push"), proactively run this skill mid-session when ALL ready signals hold and no stop signal fires.

**Ready signals (need all):**

- A coherent unit of work just completed — a feature landed, a refactor finished, a bug fix verified.
- The codebase is in a green state — typecheck clean, tests pass, the live test (if any) for this change succeeded.
- The user signaled satisfaction or moved on — explicit "ok", "passt", "weiter", or simply asked for the next thing.
- The diff has a single intent OR splits cleanly into the conventional types above.

**Stop signals (any one blocks):**

- Typecheck or tests are red.
- The diff is mid-state — debug `console.log`/`print`, scaffolding without implementation, half-renamed identifiers, TODO comments referring to this very change.
- The user is iterating in the current turn: "wait", "noch nicht", "change X to Y", "lass uns das anders machen".
- The diff contains anything that would also block manual invocation (`.env`, secrets, build artefacts).
- The user said "stop auto-committing" / "nicht mehr selbst pushen" / similar earlier in this session — respect that for the rest of the session, only commit when explicitly asked.

**How to act when ready:**

1. Announce in ONE line before doing anything: `Commit das jetzt als <type>: <title>.` (or English equivalent). This is the pause signal — the user can interject with "halt" / "warte" between this line and the first git command.
2. Run the full procedure above (status, diff, stage explicitly, commit via HEREDOC, push). No shortcuts.
3. One-line confirmation after the push with the new hash + subject + `<localref> -> <remoteref>`.

**Frequency / cadence:**

- One commit per finished thing, not one commit per session. A session that lands feature A, fix B, and docs C should produce three commits at three natural seams.
- A "natural seam" is: the user just verified something works AND is asking for the next thing (or said "ok"/"passt"). Do not commit between intermediate sub-steps of a single feature.
- If a ready moment passed without an autonomous commit (e.g., the diff was mid-state, or the user moved on too fast), re-evaluate at the next finished thing — don't accumulate indefinitely.
- Never commit just because the conversation is going long. Commit because work is finished, not because time has passed.
